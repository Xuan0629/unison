# CLAUDE.md — Unison 项目开发规范

此文件为 Claude Code 的持久化指令，在每次会话中自动加载。由 Hermes SOUL.md 派生，适配 Claude Code 的编码风格。

## 1. 先读再写

写代码前必须：
- 读要改的文件（不是浏览，是读）
- 看项目里类似功能怎么做的
- 检查 imports，别引入项目不用的库
- 看测试文件——它告诉你真正的预期行为

## 2. 编码前先想清楚

- **声明假设。** 不说"加个认证"就默认实现。先讲清楚你假设的方案。
- **命名权衡。** 每个实现选择都有取舍，说清楚。
- **多个方案时简要呈现。** 两个，最多三个，带推荐。
- **困惑就停手。** 别用看起来合理的代码填充不理解的需求。

## 3. 简洁

写刚好解决当前问题的最少代码。不要：
- 过早抽象（一个 EmailService class 只为发一封欢迎邮件）
- 过度错误处理（不可能发生的错误不需要 try/catch）
- 不必要的可配置性（hardcode 直到有真正理由需要配置）
- 死灵活性（只有一个实现的 interface 是负债不是资产）

## 4. 精确改动

- 不要碰没被要求碰的东西
- 匹配已有代码风格（引号、命名、缩进）
- 只清理你的改动造成的死代码
- 禁止格式化——不跑 prettier，不改缩进，不重排 imports

## 5. 验证

- 修 bug 前先写复现测试
- 改动前后都跑已有测试
- 不写没价值的测试（测 constructor 设属性 = 废）
- 如果无法写测试，直接说出来

## 6. 目标驱动 (P12d)

每个任务开始前必须有明确的成功标准。模糊需求要转化为可验证的目标：
- "加个校验" → ""邮箱缺失或无效时拒绝，返回 400 并说明原因，为两种情况写测试"
- "修 bug" → "写复现测试 → 修 → 确认通过 → 确认已有测试仍通过"
- "优化性能" → "先 profile → 找到瓶颈 → 针对性修复 → 再次测量"

超过一步的任务，先列计划再执行。让用户有机会在浪费时间之前纠正。

## 7. 排错 (P12d)

出问题时不要猜，要调查：
- **完整读错误信息。** stack trace 会告诉你具体是什么问题
- **先复现。** 无法复现就无法验证修复
- **一次只改一个变量。** 改三个东西一起好了 → 不知道是哪个修的
- **不要加补丁而不理解根因。** null check 可能防止崩溃，但底层的 bug 还在
- **卡住了就说。** "我试了 X 和 Y，都不行。看到的症状是……我怀疑问题在 Z"

## 8. 依赖管理 (P12d)

加包之前先想：
- 项目里已有的东西能做到吗？已经有了 axios 就别加 node-fetch
- 标准库能做到吗？不需要 lodash 做 `Array.prototype.map`
- 这个包还维护吗？看最后 commit 日期、issue 数
- 多大？500KB 的包只用来格式化日期，不值

加依赖时说明理由。默默往 package.json 加东西不行。

## 9. 沟通 (P12d)

怎么沟通代码和代码本身一样重要：
- **说清楚做了什么和为什么。** 别只丢一段代码
- **指出隐患。** 实现了要求但觉得有问题 → 主动说
- **精确表达不确定性。** "我不确定这个库支持 streaming" 有用，"应该可以" 没用
- **不要解释用户已经知道的东西。** 匹配解释深度到用户的已有知识
- **Commit message 要具体。** "Fix bug" 没用；"Fix null pointer in user lookup when email contains uppercase chars" 有用

## 10. 常见失败模式 (P12d)

察觉到以下行为立刻停止：
- **什么都修。** 让你加一个功能，你重构半个代码库
- **过度抽象。** 为一个场景造泛化框架——抄两次再考虑抽象
- **隐形决策。** 架构选择不标注——用户应该知晓
- **乐观路径。** 只写 happy path——想想 API 返回 500、文件不存在、表单为空
- **幻觉 API。** 自信地使用不存在的 API 或已移除的参数——不确定就看文档
- **风格游离。** 按自己的偏好在项目里写代码——匹配项目，不匹配偏好
- **失控重构。** 修一个 bug → 牵连三处 → 改十五个文件 → 忘了要修什么
