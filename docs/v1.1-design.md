# Unison v1.1 Design and Release Contract

## Objective

v1.1 extends the set of constrained runtimes and supervisory capabilities without weakening v1.0 local-first, single-operator, run/project isolation, auditability, or fail-closed behavior.

The first real custom Runtime Adapter is **Crush**. It is not enabled merely by accepting `runtime: crush`: its command contract, executable preflight, capability declaration, result parsing, cancellation behavior, secret/log boundary, and real subprocess integration tests must be implemented before it becomes a valid Runtime key.

## Scope and order

1. Complete foreground recovery.
2. Add Runtime Capability Metadata and the constrained Crush adapter.
3. Make Usage/Cost provenance truthful.
4. Add a mode-specific LLM Observer.
5. Run release validation and change version/tag only after all gates have evidence.

No public/multi-user WebUI, identity federation, generic plugin ecosystem, SQLiteChannel, or Windows native-lock work enters v1.1.

## Common invariants

- Every runtime invocation is bound to one project, pipeline key, run ID, role, phase, and iteration.
- Unsupported configuration, missing executable, malformed callback/artifact, unknown liveness, or policy drift fail closed.
- Prompts, environment secrets, raw logs, and user-session state do not cross project/run boundaries.
- The existing RiskEvaluator, snapshot audit, Budget, authorization, and completion detection remain mandatory. New adapters and observers do not replace them.
- A public claim requires a real execution path and behavior test.

## 1. Foreground recovery completion

### Current boundary

`unison reconcile --pipeline PATH` verifies a completed foreground result and continues the persisted serial state machine. It never re-dispatches the completed invocation. `unison resume --pipeline PATH` is the separate explicit replacement path for a proven-dead interrupted invocation.

### Implemented behavior and remaining validation

1. **Heartbeat supervision.** Orchestrator validates matching heartbeat evidence at its documented cadence. With no valid result, stale/missing heartbeats become durable `interrupted_unverified`; it halts without killing, attaching, input injection, retry, or headless downgrade.
2. **Explicit replacement resume.** `unison resume --pipeline PATH` is permitted only after durable state and current child/process-group identity prove the old invocation is not live. It records the old invocation identity, creates a new identity, checkpoints before launch, and launches only the replacement foreground invocation.
3. **Release validation remains.** Run real visible-terminal native-approval E2E for Claude and Codex in a disposable Linux repository. Implement and externally execute the macOS Terminal.app validation pack before release.

### Never

- Auto-approve native prompts, send terminal input, kill an uncertain child, retry foreground failure, or silently fall back to headless.
- Treat unreadable/mismatched PID identity as dead.

## 2. Runtime capability metadata and Crush adapter

### Capability contract

Each registered runtime exposes immutable metadata used by the loader, CLI, and WebUI. Minimum fields:

- `runtime_key`;
- supported prompt transports and whether they preserve interactive TTY;
- model override and reasoning-effort support;
- streaming and structured-result guarantees;
- usage provenance support: `actual`, `estimated`, or `unavailable`;
- session/resume support;
- executable preflight requirements;
- safe execution modes and concurrency constraints.

Capability metadata is descriptive, not authorization. A capability cannot bypass policy validation, snapshot/risk audit, Budget, or secret masking.

### Crush adapter v1.1 boundary

Crush is the first adapter only after a source-verified CLI contract is recorded. The implementation must:

- build argv arrays only; no shell interpolation;
- use a fixed prompt transport explicitly supported by Crush;
- validate the executable in dry-run and report version/contract mismatch fail-closed;
- capture stdout, stderr, exit code and log them through existing masking boundaries;
- preserve process-group cancellation and timeout semantics;
- return existing `AgentResult` semantics and enter existing completion/risk/snapshot paths;
- declare only capabilities proven by integration tests.

No generic arbitrary-command adapter is in scope. Aider or an internal CLI remains a future adapter that must independently meet the same contract.

### Bounded agent execution profiles

Implemented profiles are reusable YAML mappings that may set only `system_prompt_path`, `model`, `reasoning_effort`, `skills`, and `toolsets`. A profile and its agent cannot set the same execution field, unknown profile fields are rejected, and direct agent-level `skills` or `toolsets` are rejected.

`skills` and `toolsets` are supported only for the Hermes runtime because `HermesRunner` passes them through verified `hermes chat --skills` and `--toolsets` flags. An unprofiled Hermes agent preserves the existing default six-skill preload. Profiles cannot supply commands, environment variables, memory/session scope, provider base URLs, permissions, runtime adapters, or control authority.

## 3. Usage and cost provenance

Implemented provenance states:

- `actual`: values came from a validated runtime callback/result;
- `estimated`: values came from documented local estimation;
- `unavailable`: runtime did not supply trustworthy data and no estimate is appropriate.

OpenClaw structured output supplies actual token dimensions only when input, output, cache-read, and total values are complete and internally consistent. Current CLI runtimes use explicitly marked local estimates where their registry contract declares estimation; unavailable callbacks remain unavailable. Budget behavior remains conservative and independently enforceable.

The selected run's WebUI budget payload exposes phase-level provenance. Cost remains unavailable until a runtime returns validated cost data.

## 4. Mode-specific LLM Observer

### Current implementation boundary

- A user-interactive foreground run rejects `llm_observer`; the local operator is the approval/progress observer.
- An automated or partially automated pipeline may opt in explicitly through `llm_observer` configuration. The only accepted bindings are `runtime: hermes` and the explicit fallback `runtime: claude`; `allow_halt` and `allow_redirect` default to `false`.
- Opt-in creates a new run-bound allowlisted manifest, executes one independent no-tool observation, and writes append-only audit metadata plus a bounded report. It never reuses the current Hermes conversation/session.
- Hermes is the primary binding. It requires explicit `provider` and `model`, then invokes only `hermes -z <manifest-prompt> --provider <provider> --toolsets none --ignore-rules -m <model>`. It never uses `hermes chat` or `--yolo`; `--toolsets none` is an explicit zero-tool allowlist. The manifest is embedded only after its on-disk digest is rechecked. Hermes stdout is locally bounded to 240 characters and is persisted solely as `{status: "observed", summary}`; free-form text has no control interpretation.
- Claude remains an explicit fallback binding, proven locally with `claude -p --bare --no-session-persistence --permission-mode plan --tools "" --output-format json --json-schema ...`. Its manifest is likewise re-hashed before invocation and it accepts only bounded schema output.
- Both bindings are report-only unless a pipeline explicitly enables Claude typed control. Hermes remains permanently report-only because its binding emits unconstrained text. Claude control uses a separate strict JSON schema with identity binding (`project_id`, `pipeline_key`, `run_id`, `phase`, `iteration`, manifest SHA-256), a fixed action/reason/evidence-id vocabulary, and zero tools/session persistence. It validates every field locally before writing a bounded proposal.
- The typed-control prompt supplies the already-computed manifest SHA-256 as an exact required value; the manifest intentionally does not self-contain that digest. The independent typed-control cap is `$0.10`; the report-only Claude cap remains `$0.05`. A real isolated Claude call has produced an authorized `halt` for failed risk evidence, and a real serial Orchestrator boundary consumed it by writing its receipt before halting without dispatching an agent. Replaying the same boundary halted on the existing receipt before a second model call.
- A valid Claude proposal is consumed only at an agent-dispatch boundary. `halt` calls the existing durable `halt()` before the next agent starts; `redirect` can target only one YAML-listed `developer` and injects only a fixed Unison-compiled directive with manifest evidence IDs. No LLM free text reaches an agent prompt. A run-scoped, digest-keyed receipt is written before consumption; an existing receipt halts fail-closed rather than replaying.
- Typed control authority is rejected for foreground, MoA, chain, DAG, and parallel-development dispatch. This keeps every permitted agent route behind the common serial dispatch gate; report-only observation remains available under its existing policy.

### Implemented authority boundary

Claude typed control is limited to the fixed `halt` and `redirect` actions. `halt` requires a matching reviewer finding, failed verification, or failed risk evidence. `redirect` requires exactly one YAML-listed developer plus a YAML-listed directive code: `address_open_checklist`, `address_reviewer_findings`, or `run_declared_verification`. The evidence IDs must appear in the fresh, redacted manifest for the current run/phase/iteration.

Proposal metadata and consumption receipts never contain prompts, raw stdout/stderr, session identifiers, review text, checklist titles, or commands. An invalid, stale, digest-mismatched, cross-run, phase-mismatched, replayed, malformed, or policy-mismatched proposal is rejected and leaves the pipeline report-only. A duplicate receipt fails closed with a halt rather than repeating an action.


The LLM receives an allowlisted projection only: original goal/approved plan, current state, phase/role/iteration, selected artifact summaries, reviewer verdicts, risk and budget conclusions, and redacted event/log summaries. It cannot receive raw credentials, ambient user chat context, unbounded filesystem content, or another project's artifacts.

### Authority exclusions

Typed control never approves foreground prompts, injects terminal input, invokes reconcile/resume, kills processes, bypasses RiskEvaluator/snapshot/Budget/authorization, alters runtime configuration, or auto-reruns/replaces work. LLM output never proves liveness, completion, or process state; deterministic foreground and rule-based checks retain those responsibilities.

### Audit and failure behavior

Every invocation, context manifest digest, model/runtime selection, observation, proposed action, authorized action, rejection, timeout, and failure is append-only/run-scoped/auditable. LLM unavailability, malformed output, or context verification failure downgrades to the rule-based Observer and notification; it never unlocks more control authority.

## Verification and release gates

Each implementation slice requires a RED test, minimal implementation, focused regression, compile/diff checks, and independent five-axis review: correctness, factual honesty, isolation, regression, and scope.

Release requires:

- focused and complete repository test evidence, with unrelated pre-existing failures separately documented;
- real Linux native approval tests for Claude and Codex;
- external macOS validation evidence or an explicit release blocker;
- real Crush subprocess integration evidence;
- real Claude typed-control proposal and safe-boundary consumption evidence when typed authority is enabled;
- capability, usage provenance, Observer authorization, and cross-project isolation tests;
- documentation, package version, tag, and release notes updated only after the preceding evidence is complete.
